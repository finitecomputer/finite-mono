# Fedimint Monorepo Structure Analysis

Analysis date: 2026-07-06

Source snapshot: [`fedimint/fedimint` at `be854220fccffa2e4830b39e16a93cc29f131c9d`](https://github.com/fedimint/fedimint/tree/be854220fccffa2e4830b39e16a93cc29f131c9d)

This note is a prereq for Finite monorepo planning. It documents how Fedimint
structures a large Rust and Nix monorepo, what appears to be load-bearing, and
which patterns are worth considering for Finite.

## High-Level Takeaways

Fedimint is organized around one explicit Cargo workspace, one canonical Nix
flake, and one root command surface through `just`. Rust crates are the primary
unit of architecture, while Nix is the primary unit of environment, CI, release,
container, and package reproducibility.

The repo does not use a generic `crates/` folder. Most core packages live as
top-level `fedimint-*` directories. Larger repeated domains get domain folders:
`modules/`, `gateway/`, `crypto/`, and `utils/`. This keeps the high-value
packages visible at the root while still grouping families that share an
internal pattern.

The hand-written developer API is mostly in `justfile.fedimint.just` and
`scripts/`. The root `justfile` is generated by Flakebox and imports the custom
Fedimint commands. That separation keeps common Rust commands boring while
leaving project-specific service orchestration in a normal editable file.

The most transferable idea for Finite is not the exact directory names. It is
the layering:

- Cargo workspace for Rust dependency/version/lint coherence.
- Nix flake for reproducible shells, builds, tests, containers, and packages.
- `just` as the human command surface.
- `scripts/` as the implementation layer for commands too large for `just`.
- A first-class local integration harness that can start the whole product
  topology and feed CI.

## Rust Workspace

The workspace has 92 members in the analyzed snapshot:

| Group | Count | Pattern |
| --- | ---: | --- |
| Top-level member directories | 41 | Core libs, binaries, testing crates, docs crate, fuzz crate, and support utilities. |
| `modules/` | 38 | Repeated `common`, `client`, `server`, and `tests` module crates. |
| `gateway/` | 7 | Gateway client/server/common/db/UI/lightning/integration packages. |
| `crypto/` | 5 | Focused low-level crypto crates. |
| `utils/` | 1 | Small reusable utility crate. |

The root workspace manifest uses an explicit `members = [...]` list rather than
wide globs. That makes package membership reviewable in diffs and gives the
team a single place to notice when new crates enter the build, release, and
publish surface.

Shared workspace configuration is centralized:

- `[workspace.package]` defines authors, description, edition `2024`,
  homepage, license, repository, readme, and version.
- `[workspace.dependencies]` pins third-party versions and local path
  dependencies. Member crates depend on them with `{ workspace = true }`.
- `[workspace.lints.clippy]` applies workspace-wide Clippy policy.
- Cargo profiles are tuned globally. The `dev` profile keeps line-table debug
  info, sets selected performance-sensitive dependencies to higher
  optimization, and defines a separate `ci` profile that inherits from `dev`.
- Release profile settings are also centralized, including thin LTO and packed
  split debug info.

### Crate Categories

Fedimint's top-level crates are not all one kind of thing. The root mixes
several architectural layers:

- Core shared libraries: `fedimint-core`, `fedimint-client`,
  `fedimint-client-module`, `fedimint-server`, `fedimint-server-core`.
- Service binaries with library surfaces: `fedimintd`, `fedimint-cli`,
  `fedimint-dbtool`, `fedimint-recoverytool`, `fedimint-recurringd`,
  `fedimint-recurringdv2`.
- Support crates: `fedimint-build`, `fedimint-derive`, `fedimint-logging`,
  `fedimint-metrics`, `fedimint-bitcoind`, `fedimint-eventlog`,
  `fedimint-rocksdb`, `fedimint-db-locked`.
- Integration and testing support: `devimint`, `fedimint-testing`,
  `fedimint-testing-core`, `fedimint-server-tests`, `fedimint-wasm-tests`,
  `fuzz`.
- UI and docs support: `fedimint-server-ui`, `fedimint-ui-common`, `docs`.

Most crates declare only package-specific metadata plus dependencies. They pull
edition, license, version, repository, and many dependencies from the workspace.
That keeps individual manifests short while still letting each crate make
explicit choices about features, build dependencies, binary names, and publish
status.

### Module Pattern

The `modules/` directory is the clearest repeated package taxonomy. A module
family is generally split by runtime role:

- `fedimint-<module>-common`: shared data types, encoding, errors, endpoint
  constants, and interfaces.
- `fedimint-<module>-client`: client-side module implementation.
- `fedimint-<module>-server`: server-side module implementation.
- `fedimint-<module>-tests`: integration tests for that module.

Examples include:

- `fedimint-mint-common`, `fedimint-mint-client`,
  `fedimint-mint-server`, `fedimint-mint-tests`.
- `fedimint-wallet-common`, `fedimint-wallet-client`,
  `fedimint-wallet-server`, `fedimint-wallet-tests`.
- `fedimint-ln-common`, `fedimint-ln-client`, `fedimint-ln-server`,
  `fedimint-ln-tests`.

The repo keeps v1 and v2 versions as separate crate families:

- `fedimint-mint-*` and `fedimint-mintv2-*`.
- `fedimint-wallet-*` and `fedimint-walletv2-*`.
- `fedimint-ln-*` and `fedimint-lnv2-*`.

That is a deliberate compatibility pattern. New module generations can coexist
with old ones, be tested in mixed-version matrices, and be wired into daemon
binaries independently.

There are also `dummy`, `empty`, and `unknown` module crates. These are useful
for testing the module system and compatibility behavior without tying every
test to a production module.

### Gateway Pattern

The `gateway/` directory has its own family of crates:

- `fedimint-gateway-common`.
- `fedimint-gateway-server-db`.
- `fedimint-gateway-server`.
- `fedimint-gateway-client`.
- `fedimint-gateway-ui`.
- `fedimint-lightning`.
- `integration_tests`.

This mirrors the module split but is scoped to a larger service boundary. The
gateway is not hidden inside a generic services folder; it gets a domain folder
because it has enough internal subpackages and tests to justify it.

### Publish Boundary

Most workspace crates are publishable by default. A smaller set is explicitly
not published by setting `publish = []`: docs, fuzz/tests, gateway integration
tests, wasm tests, and module test crates. CI includes a crate ownership check
that queries crates.io for publishable workspace crates, which suggests they
treat crate publication as an operationally important surface.

## Nix Structure

Fedimint's `flake.nix` is the canonical environment and build interface.
Important responsibilities:

- Pins `nixpkgs`, `nixpkgs-unstable`, `fenix`, `flakebox`, `cargo-deluxe`,
  `bundlers`, `wild`, and the RustSec advisory database.
- Exposes overlays under `overlays`.
- Exposes a `fedimintd` NixOS module under `nixosModules`.
- Defines developer shells: `default`, `fuzz`, `lint`, `linkcheck`,
  `crossWasm`, `replit`, `bootstrap`, and `cross` on supported systems.
- Defines packages such as `fedimintd`, `fedimint-cli`, `fedimint-dbtool`,
  `gatewayd`, `gateway-cli`, `devimint`, recurring daemons, and the wasm
  bundle.
- Sets Cachix substituter configuration through `nixConfig`.

The custom Nix build layer lives in
[`nix/flakebox.nix`](https://github.com/fedimint/fedimint/blob/be854220fccffa2e4830b39e16a93cc29f131c9d/nix/flakebox.nix).
It builds on Flakebox and Crane, then adds Fedimint-specific behavior:

- Source filters split dependency-build inputs, workspace-build inputs, test
  inputs, and audit inputs. This reduces cache invalidation when unrelated
  files change.
- Common environment variables cover protobuf, RocksDB/SQLite/SQLCipher static
  linking, wasm flags, target dirs, and CI behavior.
- Workspace outputs are split into dependency builds, full workspace builds,
  doc builds, Clippy, nextest, cargo-audit, cargo-deny, coverage, feature
  checks, wasm checks, fuzz checks, and service integration tests.
- Package groups build related binaries together, then expose individual binary
  packages with a small `pickBinary` derivation.
- Container images are built with `dockerTools.buildLayeredImage` from the same
  Nix-built package groups.
- A git hash placeholder is embedded during pure builds and replaced later so
  binaries still report a meaningful revision.

The `nix/` directory is not just a flake dumping ground:

- `nix/overlays/` contains pinned or patched packages such as wasm-bindgen,
  cargo-nextest, cargo-honggfuzz, esplora-electrs, trustedcoin, and Darwin
  fixes.
- `nix/pkgs/` contains package definitions for custom dependencies.
- `nix/modules/` contains the reusable `fedimintd` NixOS module.
- `nix/tests/` contains a NixOS VM test for the module.
- `nix/bin/` contains helper scripts used by Nix builds.

The compatibility files `default.nix` and `shell.nix` forward flake behavior
through flake-compat, which keeps older Nix entrypoints working.

## `just` Command Surface

Fedimint uses two justfiles:

- `justfile`: generated from Flakebox configuration. It imports
  `justfile.fedimint.just` and defines common commands such as `build`,
  `check`, `test`, `format`, `lint`, `watch`, `clippy`, `clippy-fix`,
  `semgrep`, and `typos`.
- `justfile.fedimint.just`: hand-written project commands.

The custom justfile is the human-facing command catalog. It includes:

- Test entrypoints: `test-real`, `test-ci-all`, `test-compatibility`,
  `test-full-compatibility`, `test-upgrades`, `test-db-migrations`.
- Release readiness: `final-lint`, `final-check`, `final-check-ci`,
  `final-check-release`.
- Wasm and feature checks: `check-wasm`, `check-packages-features`,
  `check-packages-features-using-nix`.
- Local topology: `mprocs`, `devimint-env`, `devimint-env-pre-dkg`,
  `devimint-env-pre-restore`, `devimint-env-post-dkg`,
  `devimint-env-tmux`, `devimint-env-mprocs`.
- Fuzzing: `fuzz-target`, `fuzz-ci-quick`, `fuzz-target-debug`.
- Docs: `build-docs`, `build-docs-nix`, `docs`.
- Cargo manifest maintenance: `cargo-sort-fix`, `cargo-sort-check`,
  `bump-version`, `publish-release`.
- Operations and release helpers: `deploy-docker-demo`, `backport-pr`,
  `sign-release`.

The notable pattern is that `just` rarely contains large logic directly. It
usually calls a script in `scripts/` or a Nix output. This makes recipes easy to
discover while keeping complex behavior testable and reusable.

## Scripts

Fedimint's `scripts/` directory is organized by workflow:

- `scripts/dev/`: local shell aliases, docs build, devimint environment,
  mprocs launcher, test wrappers, completions, bootstrap.
- `scripts/tests/`: service-level integration tests and compatibility tests.
- `scripts/ci/`: CI-specific checks, including crate ownership.
- `scripts/ops/`: operational helpers.
- `scripts/release/`: release signing.
- `scripts/_common.sh`: shared shell library.

`scripts/_common.sh` centralizes repo-root discovery, target directory
variables, profile defaults, workspace build helpers, dev-shell checks, version
matrix helpers, and functions for using Nix-built binaries from older release
tags. This file is important because it prevents every test and dev script from
inventing its own idea of where the repo, target dir, and binaries live.

The largest test entrypoint is `scripts/tests/test-ci-all.sh`. Its shape:

- Source common helpers and normalize environment.
- Build the workspace and tests once.
- Set `CARGO_DENY_COMPILATION=1` so subsequent test commands do not silently
  trigger more builds.
- Define many named test functions.
- Generate current-version or backwards-compatibility matrices.
- Expand tests across module flags and version combinations.
- Run them with GNU `parallel`, bounded by job count, load, delay, timeout, and
  failure halt settings.
- Use `fm-run-test` to wrap each test with timing, temp dirs, logs, timeouts,
  cleanup, and failure diagnostics.

This is stronger than a pile of one-off shell scripts. It is a service topology
test runner.

## Local Integration Harness

`devimint/` is a first-class Rust crate and binary for local environment
orchestration. It knows how to stand up Fedimint daemon instances, clients,
gateways, Bitcoin/LN dependencies, test data, and local ports.

Developer commands such as `just mprocs` and `just devimint-env` build the
workspace, add target binaries to `PATH`, point `FM_DEVIMINT_STATIC_DATA_DIR`
at checked-in static data, then invoke `devimint` to start a local federation
or shell.

This is probably one of the most valuable patterns for Finite. The harness is
owned by the monorepo, built with the monorepo, and used by both humans and CI.
It avoids relying on a tribal set of terminal tabs and external service setup
steps.

## CI And Release

GitHub Actions mostly delegate to Nix:

- `ci-nix.yml` installs Nix, enables Cachix, pins flake inputs, and runs
  `nix build` outputs for lint, workspace build, Clippy, docs, doc tests,
  nextest/integration tests, wasm tests, feature checks, cargo audit, cargo
  deny, cross builds, containers, release packages, and manifests.
- `docs.yml` builds `.#nightly.ci.workspaceDocExport` and deploys the result to
  GitHub Pages.
- `ci-crate-ownership.yml` runs a crates.io ownership audit for publishable
  workspace crates.
- Additional workflows cover backwards compatibility, upgrades, scheduled
  checks, link checking, backports, PR description checks, and agent workflows.

The recurring CI pattern is: define the real work once in Nix or scripts, then
make GitHub Actions call that. CI is a scheduler and credential boundary, not a
second build system.

Release packaging is also Nix-centered:

- Binary groups are built as Nix packages.
- Individual binaries are exposed as flake packages.
- Docker images are built from Nix package outputs.
- Multi-arch manifests are assembled in GitHub Actions.
- Release checksum and signature artifacts are checked into `releases/`.

## Docs

Fedimint keeps repo docs in `docs/`, including architecture, database,
deployment, debug, module, gateway, Nix CI, release process, telemetry, and
developer tutorial documents.

`docs/` is also a Cargo workspace package named `fedimint-docs` with
`publish = []`. This lets docs participate in workspace mechanics without being
a published crate.

Technical reference docs are generated through `scripts/dev/build-docs.sh`.
That script runs `cargo doc` for the workspace, excludes fuzz, enables strict
rustdoc warnings, includes private items, and copies static crypto HTML writeups
into the generated docs tree. The docs workflow publishes this Nix-built output
to `docs.fedimint.org`.

## Quality Gates

Fedimint uses several repo-wide quality conventions:

- `treefmt` for formatting orchestration.
- `rustfmt` with edition 2024 and grouped imports.
- `taplo` and `cargo sort` for Cargo manifest sorting.
- `typos` with repo-specific word allowlists.
- `cargo clippy --workspace --all-targets -- -D warnings`.
- `cargo audit` and `cargo deny`.
- `semgrep`.
- `nextest` with repo-wide timeout behavior in `.config/nextest.toml`.
- Pre-commit hooks configured through Flakebox, including custom checks for
  forbidden dependencies and Cargo sorting.

Most of these are accessible through `just lint`, `just final-check`, and Nix CI
outputs. The quality system is broad, but developers do not need to remember
every tool directly.

## Patterns Worth Considering For Finite

Fedimint is a good reference for a Rust-heavy monorepo with reproducible
infrastructure. The strongest patterns to consider:

- Put the monorepo contract at the root: `Cargo.toml`, `flake.nix`, `justfile`,
  `scripts/`, `.github/`, `docs/`.
- Use one `Cargo.lock` and one workspace dependency table for Rust packages.
- Keep crate membership explicit, at least while the repo is young.
- Split repeated domains by role when that reflects real runtime boundaries:
  `common`, `client`, `server`, `tests` is a useful pattern for protocol or
  service modules.
- Let Nix own dev shells, CI packages, release binaries, containers, and
  cross-platform toolchains.
- Keep `just` as the developer command index, not the place where large logic
  lives.
- Create a first-class local integration harness for the whole Finite stack,
  similar in spirit to `devimint`.
- Make CI call the same Nix and script entrypoints developers can run locally.
- Keep docs in the monorepo and include generated API/reference docs in the
  normal build pipeline.

## Patterns To Be Careful Copying

Some Fedimint choices fit Fedimint better than Finite:

- Fedimint is overwhelmingly Rust. Finite is likely polyglot: Rust services,
  web apps, mobile clients, Nix/NixOS modules, Kubernetes or deployment config,
  product docs, and agent runtime assets. A pure Rust package taxonomy will not
  be enough.
- Fedimint's root has many package directories. That works because nearly all
  names share the `fedimint-*` prefix. Finite may be clearer with top-level
  domain folders such as `apps/`, `crates/`, `services/`, `infra/`, `nix/`,
  `docs/`, and `tools/`, unless we intentionally want root-level packages for
  the most important binaries.
- Flakebox is doing a lot of Rust-specific work. Finite should evaluate whether
  Flakebox fits all needed languages and deployment targets before adopting it
  wholesale.
- Fedimint's Nix layer is powerful but complex. Finite should start with the
  smallest Nix interface that gives deterministic dev shells and CI builds, then
  grow toward release packaging and containers.
- Compatibility matrices are valuable but expensive. Finite should define which
  service/version boundaries actually require matrix testing before recreating
  Fedimint's full pattern.

## Candidate Finite Monorepo Primitives

Based on Fedimint's structure, a future Finite monorepo plan should probably
define these primitives explicitly:

- Workspace root contract: root README, `flake.nix`, `justfile`, `scripts/`,
  `.github/`, `docs/`, one or more language workspaces.
- Rust contract: one Cargo workspace, shared dependency table, lints, profiles,
  and clear publish/private rules.
- App/service taxonomy: where dashboards, chat clients, runtime services,
  CLIs, shared libraries, and deployment configs live.
- Nix contract: dev shells, build packages, container outputs, deployment
  artifacts, formatter/linter tools, Cachix policy, and CI output names.
- Command contract: `just build`, `just check`, `just test`, `just lint`,
  `just dev`, `just stack-up`, `just final-check`, and service-specific
  commands.
- Integration harness: a Finite equivalent of `devimint` that can start local
  Core, runner/runtime, Hermes, chat, sites/brain/search dependencies, and
  representative test state.
- Documentation contract: this docs repo's orientation role should become a
  root `docs/` role in the monorepo, with repo-specific implementation docs
  moved next to owning code.

## Open Questions For Finite Planning

- Should Finite use root-level package directories for flagship Rust binaries,
  or prefer `crates/` and `services/` for a cleaner polyglot root?
- Which current repos become first-class apps, services, crates, infra modules,
  docs, or archived history?
- Do we want Nix to build frontend/mobile artifacts too, or only Rust/services
  and dev shells at first?
- What is the minimum useful local full-stack harness for Finite?
- Which compatibility boundaries matter: runtime image versions, chat protocol
  versions, dashboard/Core API versions, Nostr/FiniteBrain data formats, or
  hosted deployment manifests?
- Which packages, if any, are intended for public publication versus private
  monorepo use?
