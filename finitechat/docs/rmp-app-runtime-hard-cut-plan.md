# RMP App Runtime Hard Cut Plan

Status: Phase 1 through Phase 5 implemented; Phase 6 next

Finite Chat should follow the same product architecture that worked in Pika:
one Rust app runtime owns product state and policy, while native apps, CLI
harnesses, and agent-facing smoke tests dispatch typed actions and render or
inspect Rust-projected state.

The hard cut is intentional. We should delete or demote duplicate product
surfaces instead of preserving compatibility for pre-release harnesses.

## Target Shape

- `FiniteChatRuntime` is the canonical app core.
- `AppAction` is the only product command language for user-visible chat
  behavior.
- `AppState` is the product projection consumed by iOS, simulator tests, CLI
  product tests, and eventually agent/dashboard surfaces.
- Low-level client/store/protocol helpers remain internal implementation detail.
- Native app code owns rendering and bounded OS capability adapters only.
- CLI product tests must exercise the same Rust runtime iOS uses through
  UniFFI.

## Phase 1: Cut Public Product CLI to the App Runtime

Status: done

Acceptance:

- `finitechat core` is removed.
- `finitechat app` can perform real product actions through
  `FiniteChatRuntime` and `AppAction`: identity/state/start/wait/open room,
  create room, create invite, scan invite/profile, submit PIN, send message,
  mark read, refresh devices.
- Tests and docs no longer direct product flows through `FiniteChatCore`.
- The lower-level `CoreState` remains available only inside Rust as the engine
  behind the app runtime and specialized bridge code.

Implementation notes:

- Removed the public `finitechat core` CLI product surface.
- `finitechat app` now routes product commands through `FiniteChatRuntime`.
- CLI app-runtime and Hermes flow tests use the same app runtime instead of the
  removed product core entrypoint.

## Phase 2: Make Runtime Dispatch Pika-Shaped

Status: done

Acceptance:

- `dispatch(AppAction)` becomes enqueue-only from native callers.
- Rust emits rev'd updates through a single update path.
- Native code applies full snapshots and drops stale revs.
- One-shot side-effect updates are explicitly listed and handled before stale
  drops if they carry material not stored in `AppState`.
- Swift no longer owns command-tail policy or product result interpretation.

Implementation notes:

- `FiniteChatRuntime.dispatch(AppAction)` is enqueue-only in the generated
  Swift API.
- Rust owns a single actor queue and emits `AppUpdate::FullState(AppState)`
  through `AppReconciler`.
- Swift installs `listenForUpdates`, applies snapshots through one rev-gated
  path, and removed its command-tail queue.
- `waitForUpdate` now uses the runtime wait-plan/SSE hint path instead of
  polling `.startRuntime`.
- CLI/tests keep Rust-only `dispatch_and_wait` helpers for deterministic
  harnesses; this is not exposed to native callers.
- No one-shot update type is needed yet because Phase 2 only publishes full
  state snapshots.

## Phase 3: Move Remaining Product Flow State into Rust

Status: done

Acceptance:

- Invite scan, PIN submission, admission wait, wrong PIN, admitted, same-room,
  and unavailable states are represented in `AppState`.
- Busy/in-flight state is Rust-owned instead of inferred from Swift flags.
- Navigation/selection is projected from Rust and native sends route actions.
- Simulator tests join through the real UI and assert the Rust state projection.

Implementation notes:

- Added `AppFlowState` to `AppState` for user-facing flow notice text, busy
  markers, scan in-flight state, active PIN submission room, and scan outcome.
- The Rust actor publishes intermediate busy snapshots for scan/PIN actions
  without advancing `rev`, then publishes the final command state with the next
  revision. Native listeners can render immediate feedback while completion
  paths still wait for the finished command.
- Invite scanning now preserves existing connected, already-pending, joining,
  and unavailable room states instead of downgrading them back to "enter PIN".
- PIN submission, admission wait, admitted, unavailable, and failed join copy is
  generated by Rust and projected through `AppState.flow`.
- Swift removed `localNoticeText`, the stored PIN-submission flag, and local
  invite/PIN message mapping. It keeps only native drafts and renders
  `state.flow`.
- Tests now build expected scan/PIN product state through `AppFlowState`, and a
  Rust listener regression asserts the busy-to-final PIN update sequence.

## Phase 4: Bring Hermes and Agent Flows onto the Same Product Runtime

Status: done

Acceptance:

- The `finitechat hermes` binary still exposes Hermes-specific JSON bridge
  commands, but product behavior routes through `FiniteChatRuntime` wherever it
  overlaps with app behavior.
- Agent invite, admission, outbound message, read receipt, attachment, and
  activity flows share app-runtime tests instead of parallel low-level tests.
- Any remaining lower-level bridge path is documented as protocol plumbing, not
  product policy.

Implementation notes:

- `finitechat hermes init` now initializes the same runtime-backed client store
  that app/iOS flows use, while still persisting the Hermes agent identity files.
- `finitechat hermes invite` creates rooms and invite URLs through
  `FiniteChatRuntime`; Hermes keeps only the CLI flags, QR/PIN presentation, and
  stored invite URL list.
- `finitechat hermes join` uses the app-runtime scan and PIN submission flow,
  then waits on the runtime room projection to reach `Connected`.
- `finitechat hermes poll` calls a Rust-only runtime bridge poll that accepts
  owned invite requests, syncs rooms, updates app projections, and returns raw
  applied events. Hermes still owns inbox dedupe, ack files, payload decoding,
  attachment materialization, and JSON response shape.
- `finitechat hermes send`, `edit`, and `recover` encode Hermes payloads but
  append them through the runtime local-outbox send path.
- `finitechat hermes activity` appends through a runtime bridge activity input,
  so room server selection, group epoch lookup, encryption, and delivery policy
  stay in core runtime code.
- The remaining low-level Hermes code is bridge plumbing: service HTTP hosting,
  plugin install files, inbox ack state, wait-hint routing, invite URL/PIN
  display, and attachment blob materialization.

## Phase 5: Cross-Surface Proof

Status: done for local real-Hermes/iOS product proof; Docker/Tinfoil evidence is
tracked in `integrations/hermes/HARDENING.md`

Acceptance:

- CLI app-runtime tests, iOS simulator tests, Docker/Hermes sidecar tests, and
  Tinfoil canary tests exercise the same user-visible state transitions.
- A CLI passing state-machine test implies the iOS app should render the same
  state, modulo native-only capability adapters.
- A failed iOS flow must be reducible to either Rust state, native rendering,
  or native capability bridge behavior without guessing.

Implementation notes:

- The iOS app, CLI app runtime, and Hermes bridge now share the same
  `FiniteChatRuntime` action/state path for invite scan, PIN submission,
  admission, message send, read receipt, and Hermes bridge sends.
- Native callers use exported `dispatchAndWait(action:)` on background tasks for
  completed product actions. The app no longer treats "some later revision" as
  proof that a requested runtime action finished.
- `runtime_tick` reloads the durable chat projection from the encrypted local
  store after sync, then reapplies the durable outbox overlay. This keeps live
  `AppState.messages` aligned with persisted decrypted messages without
  dropping failed/undelivered local bubbles.
- Timeline row identity includes first message id, last message id, and count,
  so SwiftUI/diffing updates grouped consecutive incoming messages instead of
  silently keeping a stale row.
- The invite path claims the exact PIN-bound KeyPackage uploaded by the joiner,
  and permanent poisoned pending Welcome activation failures are cleared so a
  later valid invite can recover.
- Phase 5 was proven with a visible iOS Simulator flow against
  `https://chat.finite.computer` and a local real Hermes 0.17 gateway:
  Home -> Chats -> joined invite room -> composer send -> OpenRouter/Claude
  response -> incoming bubble, with no launch automation flags and no CLI sync
  used as the UI oracle.
